# Why One HTML Engine Matters

## You have a site. You want to add R.

You already have a blog built with Hugo, Jekyll, Eleventy, or another
static site generator. You have a theme you like, a build pipeline that
works, and shortcodes you’ve written. Now you want to include some R
output — a plot, a table, maybe an interactive widget — in a post.

The standard advice is to reach for **blogdown** or **Quarto**. Both
work, but both change your relationship with your site generator.

## The two-engine problem

blogdown routes your `.Rmd` through Pandoc before Hugo sees it. Quarto
does the same. The result is two independent Markdown-to-HTML engines in
series:

    .Rmd / .qmd
      → knitr (evaluates R)
        → Pandoc (converts Markdown to HTML)
          → Hugo / Jekyll (converts Markdown to HTML again)
            → final site

Pandoc and your SSG are both Markdown-to-HTML converters, but they are
*different* converters with different rules. Each has its own opinions
about CommonMark extensions, raw HTML handling, and template features.
This creates a practical problem: **you write for two parsers at once**.

### Shortcodes break

Hugo shortcodes like `{{</* figure src="plot.svg" */>}}` must survive
Pandoc processing untouched. This means guarding them — wrapping them in
raw HTML blocks, using special escape syntax, or writing separate
shortcode versions for Pandoc-processed content and plain Markdown
content.

This isn’t a one-time setup cost. Every shortcode you write or adopt has
to account for whether the content passed through Pandoc first. Authors
working on the Hugo side must think about Pandoc escaping, which is
unnatural for someone working in Hugo’s template language.

### Two Markdown dialects in one site

With blogdown or Quarto, `.Rmd` / `.qmd` files go through Pandoc and
your SSG. Plain `.md` files go through your SSG only. The two paths
produce subtly different HTML from similar-looking Markdown. CSS and JS
from Pandoc’s template system can conflict with the SSG theme.

The result is context-switching: Pandoc-flavoured Markdown in R files,
SSG-flavoured Markdown in everything else. Similar enough to look
interchangeable, different enough to break.

## statdown: drop Pandoc, keep your SSG

statdown takes the opposite approach from Quarto. Instead of pulling the
website into R’s toolchain, it pushes R output into your SSG’s world:

    .Rmd
      → knitr (evaluates R, splices results into Markdown)
        → CommonMark .md (with raw HTML blocks for widgets)
          → Hugo / Jekyll / any SSG (the only HTML engine)
            → final site

knitr is **not** an HTML engine. It evaluates R code chunks and inserts
the results — text, tables, plot references, widget HTML — into the
Markdown source. It does not parse or transform the surrounding
Markdown. The output is plain
[CommonMark](https://spec.commonmark.org/current/) with optional raw
HTML blocks.

One system converts Markdown to HTML: your static site generator. You
write for one parser, one set of rules, one mental model.

## What this looks like in practice

### Shortcodes just work

Because statdown outputs CommonMark, SSG features pass through knitr
unchanged. Hugo shortcodes, Jekyll includes, and Eleventy shortcodes are
just text in the Markdown source — no escaping, no guarding, no special
cases.

```` markdown
---
title: "My Post"
---

Here is a plot generated by R:

```{r}
plot(cars)
```

And here is a Hugo shortcode, used normally:

{{</* youtube dQw4w9WgXcQ */>}}
````

After
[`statdown_render()`](https://andrewjstryker.github.io/statdown/reference/statdown_render.md),
the `.md` output contains the plot reference and the shortcode verbatim.
Hugo processes both. No Pandoc in between to mangle the shortcode
syntax.

### htmlwidgets

When knitr encounters an htmlwidget (e.g. a DT datatable or a leaflet
map), statdown’s custom `knit_print` method:

1.  Extracts the widget’s CSS/JS dependencies
2.  Registers them with
    [depkit](https://github.com/andrewjstryker/depkit), which
    deduplicates and copies assets
3.  Emits the widget’s HTML inline

After knitr finishes,
[`statdown_render()`](https://andrewjstryker.github.io/statdown/reference/statdown_render.md)
prepends `<link>` tags and appends `<script>` tags to the `.md` file.
These are raw HTML blocks in CommonMark — every compliant parser passes
them through to the final page unchanged.

By default, JS assets are served from a CDN (jsDelivr) with SRI
integrity hashes and automatic local fallback. This reduces hosting
burden without sacrificing reliability.

### No special cases

With one engine, there is no Pandoc Markdown vs. Hugo Markdown
distinction. Your entire site uses the same Markdown dialect: whatever
your SSG implements. R posts and plain posts are processed identically.

## When Quarto is the better choice

Quarto solves the two-engine problem differently: it eliminates the SSG.
Quarto becomes the site builder, so there *is* only one engine — but
it’s Pandoc, not Hugo. This is a valid approach when R is the center of
your site rather than an occasional guest.

Quarto also provides capabilities that statdown does not:

- **Citations and crossrefs** — Pandoc’s citation processor and
  cross-reference system are mature and powerful.
- **Multi-format output** — one source document can produce HTML, PDF,
  slides, and Word. statdown only produces CommonMark `.md`.
- **Quarto extensions** — a growing ecosystem of filters, formats, and
  shortcodes that operate at the Pandoc layer.
- **Integrated publishing** — `quarto publish` handles deployment to
  GitHub Pages, Netlify, and other hosts.

If your workflow involves academic writing, multi-format delivery, or
heavy use of Pandoc features, Quarto is the better tool.

## Summary

statdown is for bloggers who happen to use R — people who want to keep
their static site generator and add R content without a second HTML
engine in the pipeline. By outputting CommonMark and letting your SSG
handle all HTML conversion, shortcodes work without workarounds, there’s
one Markdown dialect across your whole site, and R is a guest in your
build process rather than the host.

|                       | statdown            | Quarto / blogdown                |
|-----------------------|---------------------|----------------------------------|
| Perspective           | blogger adds R      | R produces a website             |
| HTML engines          | 1 (your SSG)        | 2 (Pandoc + SSG), or Pandoc only |
| Markdown dialect      | your SSG’s          | Pandoc + SSG                     |
| SSG shortcodes        | work directly       | need escaping                    |
| Widget dependencies   | depkit (build-time) | Pandoc template + theme          |
| Multi-format output   | no                  | yes                              |
| Citations / crossrefs | no                  | yes                              |
